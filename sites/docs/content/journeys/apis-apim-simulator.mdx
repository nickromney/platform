# Learn APIs With APIM Simulator

Outcome: understand the platform API path at a high level: a browser hits a
protected app route, the app router forwards API calls to the APIM simulator,
and the simulator forwards to the backend API. This is intentionally light
touch because the standalone `apim-simulator` repo owns the deeper product
surface.

## Run

```bash
cd ~/Developer/personal/platform
make -C kubernetes/kind 900 apply AUTO_APPROVE=1
kubectl -n apim get deploy,svc,configmap
kubectl -n dev get deploy,svc | rg "subnetcalc|router"
```

Open the protected app route after SSO is ready:

```bash
make -C kubernetes/kind check-sso-e2e
open https://subnetcalc.dev.127.0.0.1.sslip.io
```

## Inspect

The APIM simulator config is a Kubernetes ConfigMap:

```bash
kubectl -n apim get configmap subnetcalc-apim-simulator-config -o yaml
kubectl -n apim logs deploy/subnetcalc-apim-simulator
```

Trace the app router:

```bash
rg -n "subnetcalc-apim-simulator|Authorization|Ocp-Apim" \
  terraform/kubernetes/apps/workloads/base/all.yaml
```

The vendored simulator metadata tells you which standalone repo revision was
copied into this platform repo:

```bash
cat apps/subnetcalc/apim-simulator.vendor.json
```

## Try One Change

Read the simulator route table and explain how dev and UAT are separated:

```bash
kubectl -n apim get cm subnetcalc-apim-simulator-config \
  -o jsonpath='{.data.config\\.json}' | jq '.routes'
```

Then inspect the backend services:

```bash
kubectl -n dev get svc subnetcalc-api
kubectl -n uat get svc subnetcalc-api
```

<Callout type="warning">
  Do not treat the vendored simulator copy as the source of truth. Make
  meaningful APIM simulator changes in the standalone `apim-simulator` repo,
  then vendor a pinned runtime subset back into this platform.
</Callout>

### Acceptance Checks

```bash
kubectl -n apim get cm subnetcalc-apim-simulator-config -o yaml
kubectl -n apim logs deploy/subnetcalc-apim-simulator | rg "subnetcalc-api"
cd apps/subnetcalc/apim-simulator && uv run pytest
```

Use the example to identify which parts of the API contract belong to APIM and
which parts remain inside the backend app.
