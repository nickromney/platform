# Makefiles

The Makefiles are the user-facing command language for the repo. They route a
technical operator into focused workflows without requiring them to remember
script paths or raw Terragrunt commands.

The root Makefile is mostly a router. Mutating stack work belongs in focused
subtrees such as `kubernetes/kind`, `kubernetes/lima`, `kubernetes/slicer`,
`docker/compose`, `apps/subnetcalc`, or `sd-wan/lima`.

## Stage Workflow Syntax

The stage Makefiles accept several forms, but the preferred syntax is:

```bash
make -C kubernetes/kind 100 plan
make -C kubernetes/kind 900 apply AUTO_APPROVE=1
make -C kubernetes/lima 900 check-health
make -C kubernetes/slicer 900 check-security
```

`mk/stage-workflow.mk` extracts:

- the first valid stage from `MAKECMDGOALS`
- the first valid action such as `plan` or `apply`
- the first valid helper such as `check-health` or `check-security`
- unknown goals, duplicate stages, and ambiguous dispatches

That is why `make 900 apply` works without requiring separate targets for every
stage/action pair.

## Target Makefiles

### Kind

Kind owns Docker daemon checks, Kind cluster lifecycle, image-cache mode,
operator overrides, split kubeconfig, and the stage apply wrapper.

Key knobs:

```bash
KIND_IMAGE_DISTRIBUTION_MODE=registry
KIND_PRELOAD_IMAGES_MODE=off
KIND_WORKER_COUNT=1
MERGE_KUBECONFIG_TO_DEFAULT=0
```

### Lima

Lima owns VM startup, k3s bootstrap, host gateway proxy helpers, local image
cache, and the shared stage apply path.

```bash
LIMA_HOST_GATEWAY_PROXY_MODE=auto
PLATFORM_LOCAL_IMAGE_CACHE_MODE=auto
PLATFORM_BUILD_LOCAL_WORKLOAD_IMAGES_MODE=auto
```

### Slicer

Slicer owns daemon startup, VM sizing, k3s bootstrap, host forwards, profile
selection, and the Slicer-specific cache surface.

```bash
SLICER_NETWORK_PROFILE=cilium
SLICER_HOST_FORWARDS_MODE=auto
SLICER_VM_MIN_DISK_GB=25
```

### SD-WAN Lima

The SD-WAN Makefile is not part of the Kubernetes stage ladder. It owns a
three-VM Lima lab with explicit `up`, `down`, `destroy`, `test`, `logs`, and
`show-urls` targets.

## Makefile Style Tips

- Keep root targets informational unless they are repo-wide hygiene tasks.
- Put mutating workflows in focused subtrees with `make -C <dir> ...`.
- Use target-local variables with safe defaults, then export only what scripts need.
- Keep help text accurate; it is the primary UI.
- Prefer mode variables (`auto|on|off`) over magic behaviour.
- Validate mode variables early with `$(error ...)`.
- Use `AUTO_APPROVE=1` for deliberate non-interactive mutation, not as a default.
- Keep read-only checks able to flip to dry run with `DRY_RUN=1`.
- Put reusable parsing in `mk/*.mk`, not copied across targets.
- Let Make invoke scripts; do not bury complex Bash directly in recipes.

<Callout type="warning">
  Make is the operator interface, not just build automation. A target should
  explain the operational boundary it crosses: host runtime, kubeconfig,
  Terraform apply, GitOps sync, app build, or validation.
</Callout>
