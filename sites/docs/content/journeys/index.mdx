# Learning Journeys

Use these journeys when "run the stack" is too broad. Each path turns one
platform capability into a small sequence: bring up the right stage, inspect
the real objects, read the source that produced them, and make one careful
change.

## Baseline Loop

Most journeys assume the same local baseline:

```bash
cd ~/Developer/personal/platform
make -C kubernetes/kind 900 apply AUTO_APPROVE=1
make -C kubernetes/kind check-health
```

Use smaller stages when the journey says so. The smaller stages are valuable
because they let you see one capability arrive at a time:

```bash
make -C kubernetes/kind 100 apply AUTO_APPROVE=1
make -C kubernetes/kind 200 apply AUTO_APPROVE=1
make -C kubernetes/kind 300 apply AUTO_APPROVE=1
```

<Callout type="warning">
  The journeys are learning paths for the local platform workbench. Use the
  source links and checks to understand how each capability is assembled.
</Callout>

## How To Read The Pages

Each page uses the same shape:

- **Outcome:** what you should understand by the end.
- **Run:** the smallest useful command sequence.
- **Inspect:** concrete `kubectl`, `argocd`, `git`, or app commands.
- **Source:** files in `~/Developer/personal/platform` that define the behaviour.
- **Try one change:** a small exercise that teaches the boundary.
- **Closure slice:** the concrete artifact or follow-on exercise to inspect.
