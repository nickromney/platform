## The Short Version

The platform repo is a local platform workbench. Its primary outcome is a useful
Kubernetes-backed developer platform on a laptop or workstation: cluster,
Cilium, Hubble, Argo CD, Gitea, policies, app repos, Prometheus, Grafana,
VictoriaLogs, gateway TLS, and SSO.

It is not pretending to be a complete enterprise platform product yet. The
current stack covers much of the platform shape: application configuration,
infrastructure orchestration, environment management, deployment flow, and
access control. The docs call out where the implementation is complete, where it
is deliberately local, and where a production platform would need stronger
self-service, governance, and multi-tenant controls.

The product posture is source available, not contribution-led. The source repo
has its own contribution guidance for people who need to inspect that project;
this docs site is written as the product documentation surface.

The default path is [Kind](./tutorials/first-kind-stack). [Lima](./tutorials/lima-walkthrough)
and [Slicer](./tutorials/slicer-walkthrough) use the same staged platform shape
but change the host runtime underneath it. The sample apps exist to prove the
platform path rather than to be the main product.

This site treats the source repo's Makefiles, stage tfvars, shell entrypoints,
and policy trees as the contract. Older research notes and implementation plans
in the source repo are useful evidence, but they are not stronger than the code
that currently runs.

## Platform Coverage

The platform is best described as an inspectable local platform workbench. Its
main operating concerns map to this repo like this:

- **Application configuration management:** Helm values, chart conventions, app
  repos, and templated manifests show how app intent becomes Kubernetes objects.
- **Infrastructure orchestration:** Make, shell scripts, Terragrunt/OpenTofu,
  Kind, Lima, Slicer, Cilium, Gateway API, and Argo CD assemble the platform in
  explicit stages.
- **Environment management:** The local stacks give repeatable environments for
  learning and testing, with Kind as the default and Lima/Slicer as alternate
  host-runtime paths.
- **Deployment management:** Gitea and Argo CD demonstrate GitOps delivery
  rather than ad hoc `kubectl apply` workflows.
- **Role-based access control:** Stage `900` now proves the local IDP journey
  with Keycloak users, groups, Argo CD roles, Kubernetes API RBAC, Headlamp
  authorization, and app/environment route groups. Enterprise tenant lifecycle
  and governance remain outside the local workbench boundary.

Generated diagrams belong in this repo's source tree and build pipeline, not
as one-off screenshots. D2 source lives with the docs site, and rendered SVGs
are regenerated from that source when the platform model changes.

## Where To Go

- [Run the Kind stack](./tutorials/first-kind-stack) if this is your first local platform run.
- [Pick a reader path](./concepts/reader-paths) if you want the experienced-engineer TLDR or a beginner route.
- [Read the contracts](./reference/contracts) before changing Makefiles, stage defaults, GitOps, identity, routes, policy, or docs assets.
- [Check the footguns](./operations/footguns) before debugging a failed apply or smoke check.
- [Choose a learning journey](./journeys) if you want to learn one platform capability at a time.
- [Use the full Kind walkthrough](./tutorials/kind-walkthrough) for the reference stack details.
- [Use the Lima walkthrough](./tutorials/lima-walkthrough) or [Slicer walkthrough](./tutorials/slicer-walkthrough) when you are changing host runtimes.
- [Use the SD-WAN Lima lab](./tutorials/sd-wan-lima-walkthrough) for the separate overlapping-address teaching lab.
- [Build and use containers](./tutorials/container-workflow) when you are working on images or compose flows.
- [Read the mental model](./concepts/mental-model) before changing the repo structure.
- [Compare platform pathways](./concepts/platform-pathways) before enabling heavier observability or identity options.
- [Use version management](./concepts/version-management) before bumping Kubernetes, Helm charts, providers, images, or app dependencies.
- [Use the daily loop](./operations/daily-loop) when you already have a stack and need practical commands.
- [Read hardening choices](./security/hardening-choices) before changing exposure, policy, or container runtime settings.
- [Check the command reference](./reference/commands) when you know the task but not the exact target.
- [Use the source cleanup plan](./reference/source-repo-cleanup-plan) for the follow-up pruning pass in `platform`.

<Callout type="warning">
  The stack binds operator-facing services to localhost by default. Treat any
  off-host exposure as a separate security exercise, not a harmless port change.
</Callout>
