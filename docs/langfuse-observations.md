# Langfuse Observations

## 2026-05-21

- The current Langfuse Kubernetes documentation says the chart deploys Langfuse plus PostgreSQL, ClickHouse, and Redis-compatible storage by default, but the chart README and values now route those dependency images through `bitnamilegacy/*` because of Bitnami registry changes. For this platform, that makes the default subchart path a poor fit; a lighter GitOps-managed manifest set with explicit images avoids gated Bitnami legacy dependencies.
- The upstream Docker Compose file is a clearer operational inventory than the default Helm values for this use case: web, worker, Postgres, ClickHouse, MinIO, and Redis are the required moving parts for a self-contained Langfuse v3 deployment.
- DHI image availability is uneven for Langfuse: `dhi.io/langfuse:3`, `dhi.io/postgres:17.6-debian13`, and `dhi.io/redis:8.2.3-debian13` resolve, but `dhi.io/langfuse-worker:3` does not. That means the worker either needs to run from the DHI Langfuse image with an alternate command, or fall back to the official `docker.io/langfuse/langfuse-worker:3` image while keeping all dependency images off Bitnami legacy.
- The DHI Langfuse image is web-only in practice: it starts `node ./web/server.js` on port 3000 and has no `worker` directory. The worker needs the official `docker.io/langfuse/langfuse-worker:3` image unless a separate hardened worker image is published.
- DHI does not currently resolve `dhi.io/minio:latest` or `dhi.io/clickhouse:25.5.6-debian13`. The upstream compose file uses `cgr.dev/chainguard/minio` for object storage and `docker.io/clickhouse/clickhouse-server` for ClickHouse; those are acceptable non-Bitnami choices for the lightweight in-cluster dependency slice.
- Runtime user metadata matters for hardened manifests: DHI Redis declares user `nonroot`, DHI Postgres declares user `postgres`, DHI Langfuse web declares UID `65532`, the official Langfuse worker declares user `expressjs`, and Chainguard MinIO declares UID `65532`. The manifests should respect those image defaults where possible and only set pod/group-level filesystem ownership for PVCs.
- Langfuse v3 exposes different health endpoints per app container: the web container supports `/api/public/health` and `/api/public/ready`, while the worker supports `/api/health` on port 3030. The initial manifests use the health endpoints for probes because they are documented and less brittle for first boot on a constrained local cluster.
- A new Langfuse stage is not only a tfvars file. The platform exposes stages through Make help, the shared stage ladder, per-variant `variant.json` contracts, workflow `options.json`, the TUI fallback metadata, and Slicer's alternate default-CNI ladder. Stage `920` needs all of those surfaces to agree or the workflow UI/CLI can hide a valid stack shape even though `make -C kubernetes/kind 920 plan` works.
- Re-applying the shared OAuth2 proxy Argo `Application` resource with client-side apply can fail on existing `Application` CRs with `metadata.resourceVersion: must be specified for an update`. The Langfuse route added another instance to the same `for_each`, so the shared OAuth2 proxy `Application` resource should use server-side apply rather than client-side last-applied patches.
- Argo CD server-side apply validates `Secret.stringData` through typed patch construction. An unquoted all-numeric `encryption-key` is parsed as a number and fails with `expected string`; fixed local manifests should quote numeric-looking secret strings, especially the 64-character Langfuse encryption key.
- Host-side `docker manifest inspect` is not enough proof that an image works in the local cluster. Kind's kubelet failed to pull `dhi.io/langfuse:3`, `dhi.io/postgres:17.6-debian13`, and `dhi.io/redis:8.2.3-debian13` with `not found`, so the portable local stage should use public Docker Hub Langfuse/Postgres/Redis refs for now and keep the DHI finding documented rather than encoded as a runtime dependency.
- The post-OIDC health check originally hard-coded the stage-900 tfvars file. Once stage `920` adds Langfuse, that made newly created `langfuse` and `oauth2-proxy-langfuse` apps look unexpected during the same apply. The health check needs the current Kind stage tfvars, not only the original SSO bootstrap stage, for later stage additions.
- `docker.io/redis:8.2.3-alpine` pulls in Kind, but its image config has no default non-root user, so kubelet rejects it when `runAsNonRoot: true` is set without an explicit UID. The image contains a `redis` account at UID `999` and GID `1000`; the local manifest needs to set `runAsUser: 999`, `runAsGroup: 1000`, and matching PVC `fsGroup`.
- Updating the Redis StatefulSet template was not enough to clear an existing `CreateContainerConfigError` ordinal. The live StatefulSet showed the corrected template and a newer `updateRevision`, but `langfuse-redis-0` was still pinned to the old controller revision until the pod was deleted and recreated from the updated template.
- Langfuse web runs Prisma migrations at startup and exits if Postgres is not reachable at that moment. The worker also keeps noisy Redis timeout state if it starts before Redis is healthy. After dependency pods recover, web and worker may need a rollout restart to boot cleanly against the now-ready Postgres and Redis services.
- The `langfuse` namespace is labeled for Kyverno isolation, so it receives a generated `default-deny` NetworkPolicy for both ingress and egress. A self-contained Langfuse manifest still needs an explicit allow policy for web/worker traffic to Postgres, Redis, ClickHouse, and MinIO, plus SSO proxy ingress to the web service; otherwise the pods look healthy individually while the app fails with dependency timeouts.
- Service-to-service allow rules are not enough under Kubernetes `NetworkPolicy` default-deny. A `curlimages/curl` smoke pod in the `langfuse` namespace timed out resolving `langfuse-postgres.langfuse.svc.cluster.local` and `langfuse-redis.langfuse.svc.cluster.local`, so the Langfuse runtime policy also needs explicit TCP/UDP 53 egress to `kube-system` CoreDNS pods.
- With server-side apply, the Langfuse StatefulSets can apply successfully and still remain Argo `OutOfSync`. The live API injects PVC-template fields such as `apiVersion`, `kind`, `spec.volumeMode`, and a nested `status.phase` under `spec.volumeClaimTemplates`; Argo then repeatedly self-heals the same StatefulSets even though all pods are Healthy. The Langfuse `Application` should ignore only those Kubernetes-injected StatefulSet volumeClaimTemplate fields and set `RespectIgnoreDifferences=true` so meaningful spec drift is still visible.
- The SSO-protected admin hostname is correct for browser access, but SDK ingestion should use the in-cluster service unless a separate API route is deliberately added. A smoke call from a `curlimages/curl` pod to `http://langfuse-web:3000/api/public/ingestion` with the seeded `pk-lf-local-platform` / `sk-lf-local-platform` keys returned `201`, proving cluster-local apps can write traces without crossing the OAuth2 proxy.
- A pre-auth `302` from `https://langfuse.admin...` only proves the gateway can reach oauth2-proxy. The real browser path must complete OIDC and then let oauth2-proxy reach `langfuse-web.langfuse:3000`; the live 502 was caused by `sso` namespace default-deny egress, not by Langfuse health or the HTTPRoute.
- Current Langfuse docs still position Scores as the common object for deterministic checks, judge output, user feedback, and replay/eval pipelines. They also note that OpenTelemetry is the preferred future trace-ingestion path, while `/api/public/ingestion` remains a practical lightweight path for trace/generation/score batches in small demos.
- Gateway API `ReferenceGrant.spec.to` has a practical maximum of 16 entries in this cluster. Adding more SSO-protected backends to one grant failed validation, so SSO routes should split grants by flow or feature area before the gateway app grows past that limit.
- Langfuse ingestion needs both directions of policy opened: app-source egress to `langfuse-web:3000` and Langfuse namespace ingress from those app pods. A successful `sso` namespace curl proves the OAuth2 proxy path only; it does not prove runtime telemetry from application namespaces can write traces.
- The Langfuse ingestion endpoint returns HTTP `207` for batch-status responses, including an empty `{"successes":[],"errors":[]}` smoke payload. Demo clients should treat any 2xx response as transport success and leave per-event ingestion details visible in the returned body/logs when needed.
- Prometheus scraping for hardened app namespaces has three moving parts: pod metrics annotations, a scrape job or default pod discovery, and Cilium egress from `observability` to the app metrics port. The demos were discoverable before they were reachable, which produced `up{job="langfuse-demos"} == 0` until the observability egress rule was added.
- If both the default annotated-pod scrape job and a dedicated demo scrape job collect the same `/metrics` endpoint, unfiltered Grafana queries double-count counters. The Langfuse demo dashboard should filter on `job="langfuse-demos"` so the dedicated job is authoritative while annotations remain useful for discovery contracts.
- The local agentgateway route can be healthy at the gateway layer while returning `503` because its OpenAI-compatible backend is unavailable. The demo apps intentionally still emit failed generation observations and availability scores to Langfuse; this is useful because failure traces are part of the agentic observability surface, not just an error condition to hide.
- Prometheus `/metrics` should stay machine-readable text. If humans need to inspect metrics from a demo page, render the scrape output in an in-page panel and keep the raw `/metrics` link available; adding navigation HTML to the scrape endpoint would make the endpoint worse for Prometheus.
- In kind, Langfuse demos reach the model through `agentgateway-ai-gateway.agentgateway-system.svc.cluster.local/v1`, while agentgateway forwards to the host at `host.docker.internal:8000`. The host-side operator prerequisite is therefore `http://127.0.0.1:8000/v1`, and oMLX has an additional server-start step after the app is opened. Prereq checks should warn clearly when `/v1/models` is not reachable, but the demos can still run in deterministic fallback mode and emit useful failure traces.
- For agent demos, distinguish trace truth from operator UX: record the planner/final generation failure status in Langfuse metadata, but present a deterministic fallback answer in the app so the demo does not look like the agent itself failed after deterministic tools completed successfully.
- Stage-aware prereqs only help if recursive Make calls carry the selected stage. `make 920 apply` overrides `STAGE` inside the top-level Make process, but a plain recursive `$(MAKE) prereqs` can fall back to `STAGE=100`; pass `STAGE="$(STAGE)"` into recursive prereq/readiness calls so stage-920 checks such as the oMLX probe actually run.
- OAuth2 proxy in front of Langfuse is not the same thing as Langfuse-native SSO. The gateway can prove Keycloak admission and still leave `/auth/sign-in` showing Langfuse's own login form unless Langfuse is configured with its Keycloak provider variables and the IdP has the `/api/auth/callback/keycloak` redirect.
- The current Langfuse v3 self-hosted auth docs list Keycloak's required variables as `AUTH_KEYCLOAK_CLIENT_ID`, `AUTH_KEYCLOAK_CLIENT_SECRET`, and `AUTH_KEYCLOAK_ISSUER`; `AUTH_KEYCLOAK_ALLOW_ACCOUNT_LINKING=true` is needed for this local seeded `demo@admin.test` account because it already exists from headless initialization.
- In kind, `keycloak.127.0.0.1.sslip.io` resolves to loopback inside application pods, so app-native OIDC clients that use the public issuer need a host alias to the internal platform gateway service. Without that split-DNS patch, server-side discovery/token calls fail even though the same issuer works in the browser.
- Inbound policy on the internal gateway also matters for app-native OIDC. Langfuse could resolve the public Keycloak issuer to the internal gateway service, but OpenID discovery still timed out until `platform-gateway-hardened` allowed `langfuse` namespace ingress to the gateway on 443. The app namespace needs egress and the gateway namespace needs matching ingress.
- The demo app auth display should not read oauth2-proxy's opaque subject as the user-facing identity. In this stack `X-Forwarded-User` can be a UUID while `X-Forwarded-Email` carries `demo@dev.test`; shared `idpauth` browser helpers and header normalization should prefer email for display and stable local user ids.
- Local oMLX completions through agentgateway can exceed a five-second demo-client timeout even when `/v1/models` and the eventual `/v1/chat/completions` response are healthy. Keep the Langfuse ingestion timeout tight, but give LLM calls a larger budget so the UI does not report an LLM error merely because the local model needed a few more seconds.
- A direct host probe to `http://127.0.0.1:8000/v1/models` may return `401` even when the model server is started and the gateway route can call it successfully. Prereqs should distinguish connection failure from auth-protected reachability; only connection failure means the extra oMLX "start server" step was probably missed.
- Hardened Cilium policies need DNS egress as part of every service-to-service demo flow. Allowing demo pods to reach `agentgateway-ai-gateway.agentgateway-system.svc.cluster.local:80` is not enough if those pods cannot first resolve the service name through `kube-system/kube-dns` on TCP/UDP 53; the failure presents as an LLM timeout and never reaches agentgateway logs.
- Agentgateway is also hardened by destination ingress policy. A Langfuse demo pod needs both source egress to `agentgateway-ai-gateway` and destination ingress from `dev` pods labeled `app.kubernetes.io/part-of=langfuse-demos`; otherwise the UI shows an LLM timeout even though the demo app, Langfuse ingestion, and host oMLX route are all individually healthy.
- Langfuse's LLM connection API blocks internal service URLs unless the target service host is explicitly whitelisted. For this local stack, `LANGFUSE_LLM_CONNECTION_WHITELISTED_HOST=agentgateway-ai-gateway.agentgateway-system.svc.cluster.local` is the difference between a seeded Playground model and `Invalid baseURL: Blocked IP address detected`.
- `local-omlx` was a stale model label. The in-cluster agentgateway route advertises the host oMLX model through `/v1/models`, so demos should discover that model dynamically and only keep a deterministic fallback for traceable failure cases.
- Local Langfuse web can take more than five seconds to accept ingestion immediately after rollouts or long local model calls. The demo ingestion timeout should stay bounded, but fifteen seconds is a better teaching default because failed model generations still need to be visible as traces. Demo LLM calls should time out sooner than that on this teaching stack; waiting thirty seconds on a local model failure can crowd out the trace write that explains the failure.
