# Shell Scripts

The shell scripts are the operator glue for this repo. They keep host-specific
work visible: runtime checks, image cache setup, kubeconfig repair, local
forwards, GitOps seeding, health probes, security checks, and app smoke tests.
Terraform should own durable cluster state; shell should own work that depends
on the host running the command.

## Shared Style

Most scripts follow the same pattern:

```bash filename="script skeleton"
#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="${REPO_ROOT:-$(cd "${SCRIPT_DIR}/.." && pwd)}"

# shellcheck source=/dev/null
source "${REPO_ROOT}/scripts/lib/shell-cli.sh"

usage() {
  cat <<'EOF'
Usage: example.sh [--dry-run] [--execute]
EOF
  printf '\n%s\n' "$(shell_cli_standard_options)"
}

shell_cli_handle_standard_no_args usage "would do the thing" "$@"
```

Stylistic tips:

- Make every tracked Bash script pass ShellCheck unless there is a documented,
  local suppression.
- Keep tracked Bash compatible with Bash 3.2, because that is the stock Bash
  shipped with macOS.
- Use `set -euo pipefail` unless there is a concrete reason not to.
- Resolve `SCRIPT_DIR` and derive paths from it; do not depend on caller CWD.
- Use `--dry-run` and `--execute` for mutating scripts.
- Put environment requirements near the top with `: "${NAME:?message}"`.
- Keep script output operator-readable: one action per line, clear failures.
- Source `scripts/lib/shell-cli.sh` for standard flags and preview behaviour.
- Source `scripts/platform-env.sh` when a script needs repo `.env` values.
- Keep host lifecycle in shell; keep durable Kubernetes objects in Terraform or manifests.
- Prefer `jq`, `kubectl -o jsonpath`, and structured parsing over fragile text parsing.
- Write tests for script behaviour in `tests/*.bats` or target-local `tests/`.

## Compatibility Gates

Shell quality is not just a style preference in this repo. It is part of the
operator contract: the same scripts should work on a default macOS host and in
Linux-based automation.

Run the repo-level checks before changing script behaviour:

```bash
make lint-shell
make lint-bash32
scripts/audit-shell-scripts.sh --execute
scripts/check-bash32-compat.sh --execute
```

`scripts/audit-shell-scripts.sh` checks tracked shell scripts for repo hygiene,
standard entrypoint behaviour, and Bash 3.2 guardrails. `scripts/check-bash32-compat.sh`
scans tracked `*.sh` files for Bash 4+ constructs that macOS Bash 3.2 will not
understand.

Avoid Bash 4+ features in tracked scripts, including:

- associative arrays: `declare -A`
- namerefs: `local -n` or `declare -n`
- `mapfile` and `readarray`
- `wait -n`
- `coproc`
- `shopt -s globstar`
- case modification expansions such as `${value^^}` and `${value,,}`

When a script really needs newer shell behaviour, it should be an explicit
exception with a clear interpreter and call path. The default expectation is
Bash 3.2-compatible Bash.

<Callout type="warning">
  Avoid hiding host-sensitive work inside Terraform hooks unless the graph
  genuinely needs that edge. Port checks, VM startup, Docker cleanup, and
  browser validation are better as explicit shell targets.
</Callout>

## When Adding A New Script

1. Put it beside the target that owns the behaviour.
2. Use `--dry-run` / `--execute` for anything mutating.
3. Add a Make target instead of expecting users to remember the script path.
4. Add Bats coverage when parsing, deletion, kubeconfig, or port behaviour is involved.
5. Keep environment variables explicit and documented in the target help.
6. If the script creates cluster objects, ask whether Terraform or GitOps should own that object instead.
