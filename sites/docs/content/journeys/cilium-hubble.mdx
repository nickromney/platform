# Learn Cilium And Hubble

Outcome: understand why the platform disables the default Kind CNI, installs
Cilium at stage `200`, enables Hubble at stage `300`, and later applies Cilium
policies through GitOps.

## Run

```bash
cd ~/Developer/personal/platform
make -C kubernetes/kind 200 apply AUTO_APPROVE=1
kubectl -n kube-system get pods -l k8s-app=cilium

make -C kubernetes/kind 300 apply AUTO_APPROVE=1
kubectl -n kube-system get deploy,svc | rg "hubble|cilium"
```

After the full stack is running, validate policy:

```bash
make -C kubernetes/kind 900 apply AUTO_APPROVE=1
scripts/validate-cilium-policies.sh --mode static --execute
scripts/validate-cilium-policies.sh --mode live --execute
```

## Inspect

Cilium first:

```bash
kubectl -n kube-system get cm cilium-config -o yaml | rg "enable|hubble|wireguard"
kubectl -n kube-system rollout status daemonset/cilium
kubectl get ciliumnetworkpolicies -A
```

Hubble next:

```bash
cd terraform/kubernetes/scripts
./hubble-check-connection.sh --execute

./hubble-capture-flows.sh \
  -P --kubeconfig ~/.kube/kind-kind-local.yaml \
  --since 10m \
  --namespace dev \
  | ./hubble-summarise-flows.sh --report edges --aggregate-by workload
```

Read drops separately because they often teach more than successful flows:

```bash
./hubble-capture-flows.sh \
  -P --kubeconfig ~/.kube/kind-kind-local.yaml \
  --since 10m \
  | ./hubble-summarise-flows.sh --report drops
```

## Try One Change

Use Hubble to explain a policy before changing it. Exercise the Sentiment app,
capture flows from the `dev` namespace, and answer:

- Which workloads talk to DNS?
- Which workloads talk to the APIM namespace?
- Which flows go to `world`?
- Which denied flows are expected after policies are on?

<Callout type="warning">
  Hubble evidence is input to policy review, not automatic truth. A captured
  flow can be accidental, overly broad, or caused by a broken test run.
</Callout>

### Acceptance Checks

```bash
./terraform/kubernetes/scripts/hubble-capture-flows.sh \
  -P --kubeconfig ~/.kube/kind-kind-local.yaml \
  --since 5m --namespace dev \
  | ./terraform/kubernetes/scripts/hubble-summarise-flows.sh --report drops

scripts/validate-cilium-policies.sh --mode static --execute
scripts/validate-cilium-policies.sh --mode live --execute
```

The exercise is complete when you can point to the gateway-to-workload flow and
explain why the policy selector is not a namespace-wide allow.
