# Guided Workflows

Guided workflows are the operator-friendly layer over the staged Kubernetes
Makefiles. They keep the well-known shell commands as the source of truth, but
help you choose a target, stage, action, and app toggles without
remembering every flag.

The command flow is:

```text
Bubble Tea TUI or FastAPI/HTMX browser UI
  -> scripts/platform-workflow.sh
  -> make -C kubernetes/<target> <stage> <action>
```

The wrapper does not replace the stage Makefiles. It builds the same command
you would type by hand and, when app toggles are selected, writes a small
generated tfvars file under `.run/operator/`.

## Terminal TUI

Start the terminal UI from the repo root:

```bash
make tui
```

The TUI is a Bubble Tea app in `tools/platform-tui`. It requires Go when run
from source with `make tui`; the direct Make workflows do not require Go. Build
a local binary with:

```bash
make build-tui
```

or run the Go helper directly:

```bash
go run ./tools/platform-tui/cmd/platform-tui --execute
```

Choose a target, stage, action, and app toggles from the guided screens. A
typical plan-only smoke test is:

```text
kind
900 sso
plan
Disable sentiment
Enable subnetcalc (stage default)
```

Choose `Reset target` from the stage screen when you need the target-local reset
workflow. The preview should show `make -C kubernetes/<target> reset
AUTO_APPROVE=1`.

For a full Kind stage-900 apply without the sentiment reference app, choose the
same options but select `apply`. The preview should include a command shaped
like:

```bash
env PLATFORM_TFVARS=/Users/nickromney/Developer/personal/platform/.run/operator/kind-stage900.tfvars make -C kubernetes/kind 900 apply AUTO_APPROVE=1
```

<Callout type="warning">
  `apply` is still a real Make workflow. Use `plan` first when you are exploring
  combinations, especially when the live cluster is already at a higher stage.
</Callout>

### TUI Affordances

The TUI is the most polished local operator path. It includes:

- explicit `Back` and `Quit` items on intermediate screens;
- stage-aware app prompts, so stages `100` through `600` skip sentiment and
  subnetcalc toggles;
- streaming command output in a scrollable viewport;
- output review with `PgUp`, `PgDn`, `u`, `d`, `Home`, and `End`;
- auto-follow while output streams, with manual scrollback preserving your
  position;
- focus moving to `Back` after successful runs, so a destructive action is not
  selected again by default;
- opinionated next-stage shortcuts after successful applies, for example
  `600 apply`, `900 apply`, and Kind-only `950 apply` after `500 apply`.

## Shared Core

The non-interactive contract lives in `scripts/platform-workflow.sh`. Use it
when you want the same abstraction without a prompt:

```bash
scripts/platform-workflow.sh preview --execute \
  --target kind \
  --stage 900 \
  --action plan \
  --app sentiment=off
```

Run the generated workflow with:

```bash
scripts/platform-workflow.sh apply --execute \
  --target kind \
  --stage 900 \
  --action plan \
  --app sentiment=off
```

The supported app toggles are:

| Toggle | Effect |
| --- | --- |
| `--app sentiment=off` | Disables the sentiment reference app repo/workload. |
| `--app subnetcalc=off` | Disables the subnetcalc reference app repo/workload. |
| `--app sentiment=on` | Forces the sentiment app repo/workload on. |
| `--app subnetcalc=on` | Forces the subnetcalc app repo/workload on. |

Kind also has a first-class post-900 local IDP target. It keeps the SSO/IDP
path but trims heavier optional local services:

```bash
scripts/platform-workflow.sh preview --execute \
  --target kind \
  --stage 950-local-idp \
  --action plan
```

## Browser UI

The browser UI is a local FastAPI + HTMX wrapper over the same workflow core.
It is useful when you want a lightweight web surface without a JavaScript build
pipeline.

```bash
make workflow-ui
```

or run the helper directly:

```bash
scripts/platform-workflow-ui.sh --execute
```

Then open:

```text
https://console.127.0.0.1.sslip.io:8443
```

The UI is a small FastAPI + HTMX app under `tools/platform-workflow-ui`. It is
served through `uv` by `scripts/platform-workflow-ui.sh`, vendors HTMX locally,
and does not use Node, Vite, React, or `node_modules`.

The browser UI supports command preview, running the selected workflow, polling
command output, stage-aware app toggles, and next-stage actions after a
successful apply. It intentionally stays local and small:

```text
scripts/platform-workflow-ui.sh
  -> uv run --project tools/platform-workflow-ui
  -> hypercorn platform_workflow_ui.main:app
  -> scripts/platform-workflow.sh
```

HTTPS over HTTP/2 is the default. It uses Hypercorn and a local mkcert
certificate under `.run/workflow-ui` unless you pass explicit certificate paths
to `scripts/platform-workflow-ui.sh`. Plain HTTP is still available as an
opt-in fallback:

```bash
make workflow-ui WORKFLOW_UI_HTTP=http1 WORKFLOW_UI_HOST=127.0.0.1 WORKFLOW_UI_PORT=8765
```

Use a different HTTPS port when `8443` is already taken:

```bash
make workflow-ui WORKFLOW_UI_PORT=9876
```

<Callout type="warning">
  Keep the workflow UI on loopback addresses such as
  `console.127.0.0.1.sslip.io` or `127.0.0.1`. It is a local operator helper,
  not a hardened remotely exposed control plane.
</Callout>

## Plan Matrix

The workflow matrix covers every Kind stage from `100` to `900`, plus the
Kind-only `950-local-idp` target, at the shared workflow layer. It also covers selected app-toggle combinations
that have exposed brittleness before.

```bash
bats tests/platform-workflow.bats tests/platform-workflow-matrix.bats
```

For live plan checks against the current Kind cluster, use the shared core:

```bash
for stage in 100 200 300 400 500 600 700 800 900; do
  scripts/platform-workflow.sh apply --execute \
    --target kind \
    --stage "$stage" \
    --action plan
done
```

If your live cluster is already at stage `900`, lower-stage plans will naturally
show planned destroys. That is not a failed plan by itself. Treat the plan exit
status and the specific resources as the signal.
