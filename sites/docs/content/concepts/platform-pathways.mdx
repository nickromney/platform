# Platform Pathways

The Kubernetes stack keeps one current observability and identity path so the
local test matrix stays small.

## Observability Pathways

| Pathway | Status | Use when | Tradeoff |
| --- | --- | --- | --- |
| Prometheus + Grafana + VictoriaLogs | Current stage-800/900 path. | You want the local platform to expose metrics, dashboards, and logs while staying reasonable on a laptop. | Focused local stack with one log store. |

The current default Kind/Lima stage files set:

```hcl
enable_prometheus    = true
enable_grafana       = true
enable_victoria_logs = true
```

Retired observability variants are intentionally absent from the workflow and
stage tfvars.

## Identity Pathways

| Pathway | Status | Use when | Tradeoff |
| --- | --- | --- | --- |
| Keycloak + `oauth2-proxy` | Default Kubernetes stage-900 path. | You want org roles, app/environment groups, OIDC clients, and a real IdP administration surface. | More realistic IDP surface, more memory and slower cold starts. |
| Compose identity proof | Docker Compose only. | You want a compact browser-login proof outside Kubernetes. | Separate substrate from the Kubernetes stack. |

Stage `900` opts for Keycloak because the IDP journey now proves more than
login: realm bootstrap, users, groups, OIDC clients, Argo CD group RBAC,
Kubernetes API RBAC, and app/environment authorization.

## Operator Rule

Default pathways should be the first-run and documentation baseline. Alternative
pathways should be explicit opt-ins with memory expectations, tfvars, and a
focused validation command.

> Warning: Do not mix retired or experimental pathway toggles into the default
> stage files unless the same change also documents the memory impact and adds a
> focused validation command. Otherwise the local reset/apply proof can pass one
> operator's cache and fail on a clean laptop.

```bash
make -C kubernetes/kind 900 plan
make -C kubernetes/kind 900 apply AUTO_APPROVE=1
make -C kubernetes/kind check-app APP=grafana
```
