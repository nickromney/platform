# Review Environments

Review environments are branch previews for services created from the Portal
template. Backstage still owns the scaffolding experience internally, but the
runtime path is a platform concern: Gitea Actions builds the branch image, the
in-cluster runner applies Kubernetes resources, and the platform gateway exposes
the branch URL.

The URL shape is deterministic:

```text
https://<service>-<branch>.review.127.0.0.1.sslip.io
```

## Runtime Contract

The stage `900` substrate provides the shared pieces. The branch workflow owns
only the per-branch application objects.

| Layer | Owner | Role |
| --- | --- | --- |
| Portal template | `apps/backstage/catalog/templates/platform-service` | Generates the service repo and `.gitea/workflows/review-environment.yaml`. |
| Gitea Actions | `terraform/kubernetes/apps/gitea-actions-runner` | Runs the branch workflow on `self-hosted`, `in-cluster`, `review-env`. |
| Review namespace | `terraform/kubernetes/namespaces.tf` | Hosts branch preview Deployments, Services, HTTPRoutes, and policies. |
| Gateway TLS | `terraform/kubernetes/apps/cert-manager-config` | Issues the wildcard certificate for `*.review.127.0.0.1.sslip.io`. |

The workflow creates frontend and backend Deployments, Services,
CiliumNetworkPolicies, an HTTPRoute, and a ReferenceGrant in the shared
`review` namespace. Deleted branches trigger cleanup for objects labeled with
the matching review slug.

## When One Appears

Expect a review environment only when all of these are true:

- The service was scaffolded from the Portal platform-service template.
- A non-`main` branch was pushed to the generated Gitea repository.
- The workflow can match a runner with `self-hosted`, `in-cluster`, and
  `review-env` labels.
- The local stack is using a runner-capable image mode.
- Stage `900` has applied the `review` namespace, registry pull secret, gateway
  wildcard certificate, and policy substrate.

For Kind, the default registry image mode validates the substrate but keeps the
runner disabled. Use a runner-capable mode such as `load` or `baked` when you
need to exercise actual branch preview creation.

```bash
KIND_IMAGE_DISTRIBUTION_MODE=load make -C kubernetes/kind 900 apply AUTO_APPROVE=1
```

<Callout type="warning">
  Review environments are intentionally lightweight. Do not assume full database
  support for branch previews. Services should use in-memory state, seeded
  fixtures, or external mock data until the platform explicitly grows database
  cloning or isolated per-branch data services.
</Callout>

## Useful Checks

Use the split Kind kubeconfig when checking the local stage `900` cluster:

```bash
kubectl --kubeconfig ~/.kube/kind-kind-local.yaml --context kind-kind-local get ns review
kubectl --kubeconfig ~/.kube/kind-kind-local.yaml --context kind-kind-local -n review get secret gitea-registry-creds
kubectl --kubeconfig ~/.kube/kind-kind-local.yaml --context kind-kind-local -n platform-gateway get certificate platform-gateway-tls
kubectl --kubeconfig ~/.kube/kind-kind-local.yaml --context kind-kind-local -n gitea-runner get pods
```

If the runner namespace is empty, branch previews will not launch even though
the shared review substrate exists.
